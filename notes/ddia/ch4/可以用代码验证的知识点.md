## 优先级 High

### 1. 列式存储 vs 行式存储（Column Store vs Row Store）

**对应章节**：§ 列式存储  
**文件位置**：`practice/storage-engines/column_store.py`（待实现）

行存和列存读同一份数据时，I/O 量差多少？代码可以直接量化：

- 实现一个 `RowStore`：数据按行紧凑存储，读取时必须扫描完整行
- 实现一个 `ColumnStore`：同一列的所有值连续存储
- 用同一个 OLAP 查询（只读 3 列、聚合 100 万行）跑两个实现，打印读取字节数对比

这个验证能让"列存只读需要的列"不再是抽象说法，而是可以看到具体的字节数差。

**核心代码结构**：
```python
class RowStore:
    def scan_columns(self, col_names, predicate) -> list:
        # 必须遍历所有行的所有列，再过滤
        bytes_read = n_rows * row_width  # 浪费

class ColumnStore:
    def scan_columns(self, col_names, predicate) -> list:
        # 只读目标列的数据
        bytes_read = n_rows * len(col_names) * col_width  # 节省
```

### 2. 列压缩：Run-Length Encoding + Bitmap Encoding

**对应章节**：§ 列压缩  
**文件位置**：`practice/storage-engines/column_compression.py`（待实现）

两个压缩算法，代码可以直接演示压缩率和查询加速：

**Run-Length Encoding（RLE）**：  
排序后的列会出现大量连续重复值，RLE 把 `[31, 31, 31, 68, 68]` 变成 `[(31,3),(68,2)]`。  
代码可以打印：原始字节 vs 压缩后字节，以及按日期排序前后的压缩率差异。

**Bitmap Encoding**：  
低基数列（`product_id` 只有 10 万个不同值）用 bitmap 比存原始值节省空间，  
且多列过滤只需要 `bitmap_A AND bitmap_B`，是 bitwise 操作，非常快。  
代码可以演示：
```python
# WHERE product_sk = 31 AND store_sk = 3
result = bitmap_product[31] & bitmap_store[3]
# result 的每个 bit 代表一行是否满足两个条件
matching_rows = [i for i, bit in enumerate(result) if bit]
```

这是"为什么列存过滤快"最直观的解释，比文字描述更清楚。

### 3. Write Amplification 测量

**对应章节**：§ Write Amplification  
**文件位置**：`practice/storage-engines/write_amplification.py`（待实现）

书里说 B-Tree 写放大来自"WAL + 整页覆盖"，LSM-Tree 写放大来自"多轮 Compaction"。  
代码可以在已有的 `BTree` 和 `LSMTree` 上加一个字节计数器，实测不同 workload 的写放大：

```python
class WriteCounter:
    def __init__(self):
        self.app_bytes = 0     # 应用层写入了多少字节
        self.disk_bytes = 0    # 磁盘实际写了多少字节

write_amplification = disk_bytes / app_bytes
```

测试场景：
- 顺序写（append-only）
- 随机写（update existing keys）
- 高重复率写（反复更新同一批 key）

结果能直接验证书里的结论：LSM 在随机写 workload 下写放大通常低于 B-Tree；  
但 LSM 的 Compaction 在特定场景（大量数据、多层合并）会放大很多。

### 7. Compaction 策略对比（Size-Tiered vs Leveled）

**对应章节**：§ 压实策略  
**文件位置**：`practice/storage-engines/lsm/compaction_strategies.py`（待实现，加到 lsm 包里）

LSM 已有 Size-Tiered 实现，可以加 Leveled Compaction，然后在同一 workload 下比较：

| 指标 | Size-Tiered | Leveled |
|------|-------------|---------|
| 写放大 | 低 | 高 |

| 读放大 | 中 | 低（L1+ 无重叠）|

代码可以统计每种策略在 100 万次写入后的文件数、总磁盘字节、读取一个 key 需要查几个文件。

## 优先级 Low

### 8. Clustered Index vs Heap File

**对应章节**：§ 在索引中存储值  
**文件位置**：`practice/storage-engines/index_types.py`（待实现）

已有详细笔记（`SQL 表数据如何映射到 B-tree.md`），代码可以直接模拟两种结构：

- `HeapIndex`：B-Tree leaf 存 `key -> heap_offset`，再去 heap file 取完整 row（两次 I/O）
- `ClusteredIndex`：B-Tree leaf 直接存完整 row（一次 I/O）
- `CoveringIndex`：B-Tree leaf 存 `(key, included_cols)`，查询只含这些列时不回表

演示不同 SELECT 模式下的 I/O 次数差异。
