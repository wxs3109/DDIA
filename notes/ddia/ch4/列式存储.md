# 列式存储

`column-oriented storage` 是 DDIA 第 4 章从 OLTP storage engine 转向 analytical storage engine 时的核心概念。它解决的不是“如何快速按 primary key 找到一行”，而是“如何高效扫描很多行里的少数几列，并做过滤、join、aggregation”。

这也是为什么原书把它放在 `analytics` 语境下讲：数据仓库查询通常不是一次读写一两行，而是扫描大量 `fact table` rows，然后只用其中少数 columns 做计算。

## Row-Oriented 和 Column-Oriented 解决的问题不同

在 `row-oriented storage` 里，一行的所有 columns 被放在一起：

```text
row 1: date_key, product_sk, customer_sk, store_sk, quantity, price, ...
row 2: date_key, product_sk, customer_sk, store_sk, quantity, price, ...
row 3: date_key, product_sk, customer_sk, store_sk, quantity, price, ...
```

这种布局适合 OLTP：

```sql
SELECT * FROM orders WHERE order_id = 251;
```

因为 OLTP 查询经常是“按 primary key / secondary index 找少量 rows，然后读取或更新整行”。应用通常需要这行的大部分字段，比如订单详情页、用户 profile、一次 transaction update。把整行放在一起，读取和更新都直观。

但分析查询的模式不一样。比如原书的例子：

```sql
SELECT
    dim_date.weekday, dim_product.category,
    SUM(fact_sales.quantity) AS quantity_sold
FROM fact_sales
    JOIN dim_date ON fact_sales.date_key = dim_date.date_key
    JOIN dim_product ON fact_sales.product_sk = dim_product.product_sk
WHERE
    dim_date.year = 2024 AND
    dim_product.category IN ('Fresh fruit', 'Candy')
GROUP BY
    dim_date.weekday, dim_product.category;
```

这个查询可能扫描很多 `fact_sales` rows，但真正需要的 columns 很少：`date_key`、`product_sk`、`quantity`。如果 `fact_sales` 有 100 多列，row-oriented database 可能为了拿这 3 列，把很多完整 rows 所在的 pages 都读进内存，I/O 浪费很大。

`column-oriented storage` 的布局反过来：

```text
date_key:    row1, row2, row3, row4, ...
product_sk:  row1, row2, row3, row4, ...
quantity:    row1, row2, row3, row4, ...
price:       row1, row2, row3, row4, ...
```

这样查询只需要 `date_key`、`product_sk`、`quantity` 时，就只读取这些 columns 的数据，不需要把其他 100 多列一起读出来。

简单判断：

```text
OLTP / point lookup / frequent row update / 需要整行       -> row-oriented 更自然
OLAP / scan many rows / use few columns / aggregation      -> column-oriented 更自然
```

当然真实系统会混合：HTAP database、columnar index、covering index、materialized view、row store + column store coexist 都很常见。这里的重点是 workload shape，而不是绝对阵营。

## Column-Oriented 底层不是简单的 B-tree

读到这里容易问：column-oriented storage 底层是不是 `B-tree`？或者是不是 `SSTable`？答案是：**通常不是用 B-tree 作为主要数据布局；它更像一组 immutable column files / column chunks / row groups，再配合 metadata、encoding、compression、sometimes indexes。**

`B-tree` 适合按 key 做 point lookup 或 range lookup，例如：

```text
id -> row location
email -> primary key
```

但 column store 的主要目标是扫描和压缩。它关心的是：

```text
只读 query 需要的 columns
顺序扫描这些 columns
用 compression 减少 I/O
用 metadata 跳过不相关 blocks
用 vectorized execution 批量处理 values
```

以 Parquet / ORC 这类格式为例，可以粗略理解成：

```text
table file / object
  row group 1
    column chunk: date_key values for rows 1..N
    column chunk: product_sk values for rows 1..N
    column chunk: quantity values for rows 1..N
    metadata: min/max/null count/encoding/compression info
  row group 2
    column chunk: date_key values for rows N+1..2N
    column chunk: product_sk values for rows N+1..2N
    column chunk: quantity values for rows N+1..2N
    metadata: ...
```

所以它不是“一整列从头到尾一个巨大数组”。原书也强调：实际系统会把表拆成数千或数百万 rows 的 blocks / chunks，然后在每个 block 内分别存每个 column。

和 `SSTable` 的关系是：column store 的写入常常采用 `log-structured` 思路，生成 immutable files，然后后台 merge / compact。这一点和 SSTable / LSM tree 有相似的工程味道：旧文件不原地改，新文件批量写。但它不等于“column store 就是 SSTable”。SSTable 是 sorted key-value file；columnar file 是按 columns 分开编码的数据块，目标是 analytical scan。

## Column Layout 如何保持 Row 的关系

column store 最大的疑问是：如果每列分开存，怎么知道这些值属于同一行？

关键规则是：**每个 column 必须以同样的 row order 存储。**

例如原始 rows 是：

```text
row 1: date=2024-01-01, product=31, quantity=2
row 2: date=2024-01-01, product=68, quantity=1
row 3: date=2024-01-02, product=31, quantity=5
```

column-oriented layout 是：

```text
date_key:    [2024-01-01, 2024-01-01, 2024-01-02]
product_sk:  [31,         68,         31]
quantity:    [2,          1,          5]
```

第 `k` 个 `date_key`、第 `k` 个 `product_sk`、第 `k` 个 `quantity` 属于同一行。要重建第 2 行，就取每列的第 2 个值：

```text
date_key[2] + product_sk[2] + quantity[2]
= row 2
```

所以 column store 可以分列存储，但不能让每列独立乱序。row identity 依赖的是所有 columns 共享同一个 row position。

## 排序为什么不会搞乱 Columns

原书讲 `sort order in column storage` 时，重点就是这个坑：**不能独立排序每一列。**

如果你把每列单独排序：

```text
date_key sorted independently
product_sk sorted independently
quantity sorted independently
```

那第 `k` 个 date、第 `k` 个 product、第 `k` 个 quantity 就不再属于同一行，整张表的 row 关系被破坏了。

正确做法是：**先按 sort key 对整行排序，然后再按列拆开存。**

例如选择 `date_key` 作为第一个 sort key，`product_sk` 作为第二个 sort key：

```text
先排序完整 rows:
(date=2024-01-01, product=31, quantity=2)
(date=2024-01-01, product=68, quantity=1)
(date=2024-01-02, product=31, quantity=5)

再拆成 columns:
date_key:    [2024-01-01, 2024-01-01, 2024-01-02]
product_sk:  [31,         68,         31]
quantity:    [2,          1,          5]
```

这样所有 columns 仍然共享同一个 row order，因此 row 不会乱。

排序的好处有两个：

1. `filter pruning`：如果常按 date 查询，把 `date_key` 放在前面，查询上个月数据时可以跳过很多不相关 blocks。
2. `compression`：排序后，相同或相近 values 会连续出现，更容易压缩。

例如按 `date_key` 排序后：

```text
date_key:
2024-01-01, 2024-01-01, 2024-01-01, 2024-01-01,
2024-01-02, 2024-01-02, 2024-01-02, ...
```

这种长串重复值非常适合 `run-length encoding`。

## Column Compression：为什么列式存储更容易压缩

Column-oriented storage 通常比 row-oriented storage 更容易压缩，因为同一列里的 values 类型相同、分布相似、重复率高。

例如 `product_sk` 这一列可能有数十亿 rows，但不同 product id 只有 100,000 个。`date_key`、`store_sk`、`category` 这种列重复更明显。把同类 values 放在一起，压缩算法更容易发现规律。

原书重点讲了 `bitmap encoding`。

假设 `product_sk` 只有三个可能值：31、68、69。可以为每个 distinct value 建一个 bitmap：

```text
product_sk values by row:
row:        1  2  3  4  5  6
product:   31 68 31 69 68 31

bitmap for product=31:
            1  0  1  0  0  1

bitmap for product=68:
            0  1  0  0  1  0

bitmap for product=69:
            0  0  0  1  0  0
```

查询：

```sql
WHERE product_sk IN (31, 68, 69)
```

可以加载三个 bitmaps，然后做 bitwise `OR`。

查询：

```sql
WHERE product_sk = 30 AND store_sk = 3
```

可以分别加载 `product_sk = 30` 的 bitmap 和 `store_sk = 3` 的 bitmap，然后做 bitwise `AND`。这里的关键是：bitmap 的每一位不是一个 value，而是一个 row position。第 1 位表示第 1 行是否满足条件，第 2 位表示第 2 行是否满足条件，以此类推。

例如有 6 行销售数据：

```text
row position: 1   2   3   4   5   6
product_sk:   30  30  68  30  31  30
store_sk:     3   8   3   3   3   9
quantity:     2   1   5   4   7   1
```

现在看第一个条件：

```sql
product_sk = 30
```

满足这个条件的是 row 1、2、4、6，所以它的 bitmap 是：

```text
row position:       1  2  3  4  5  6
product_sk = 30:    1  1  0  1  0  1
```

再看第二个条件：

```sql
store_sk = 3
```

满足这个条件的是 row 1、3、4、5，所以它的 bitmap 是：

```text
row position:       1  2  3  4  5  6
store_sk = 3:       1  0  1  1  1  0
```

两个条件之间是 `AND`，意思是同一行必须同时满足两个条件。因此对两个 bitmaps 做 bitwise `AND`：

```text
row position:       1  2  3  4  5  6
product_sk = 30:    1  1  0  1  0  1
store_sk = 3:       1  0  1  1  1  0
-------------------------------------
AND result:         1  0  0  1  0  0
```

结果里只有 row 1 和 row 4 是 `1`，说明只有这两行同时满足：

```text
row 1: product_sk = 30 AND store_sk = 3
row 4: product_sk = 30 AND store_sk = 3
```

接下来 query engine 如果要计算这些 rows 的销售数量，就可以用这个 result bitmap 去 `quantity` column 里取 row 1 和 row 4 的值：

```text
quantity[1] = 2
quantity[4] = 4
SUM = 6
```

这就是“所有 columns 的 row order 一致”的意义：`product_sk` 的第 4 个值、`store_sk` 的第 4 个值、`quantity` 的第 4 个值都属于同一行。bitmap 不需要保存完整 row，只要保存每个 row position 是否满足条件，就能把多个 column 上的过滤条件组合起来。

## Run-Length Encoding 是怎么回事

你提到的 “running 编码压缩” 这里应该是 `run-length encoding`，通常简称 `RLE`。`run` 指的是一段连续重复的值。

比如一个 column 原始 values 是：

```text
A A A A A B B B C C A A
```

`run-length encoding` 会把它编码成：

```text
(A, 5), (B, 3), (C, 2), (A, 2)
```

意思是：

```text
A repeated 5 times
B repeated 3 times
C repeated 2 times
A repeated 2 times
```

对于排序后的 column，RLE 特别有效。比如按 `date_key` 排序后：

```text
2024-01-01 repeated 3,000,000 times
2024-01-02 repeated 2,800,000 times
2024-01-03 repeated 3,100,000 times
```

就可以编码成：

```text
(2024-01-01, 3000000)
(2024-01-02, 2800000)
(2024-01-03, 3100000)
```

原书在 bitmap encoding 那里也提到 bitmaps 往往很 sparse，会包含大量连续 0 或连续 1，因此 bitmaps 也可以做 RLE。`roaring bitmaps` 这类结构会根据局部数据分布在不同 bitmap 表示之间切换，选择更紧凑的形式。

RLE 的收益不是只省空间。更少 bytes 意味着更少 disk I/O 和 network I/O；有些 query engine 还能直接在压缩数据上计算，减少 decoding 成本。

## Column Store 怎么查询

Column-oriented query 的核心流程是：只加载需要的 columns，然后用 shared row positions 把不同 columns 的条件组合起来。

以上面的分析查询为例，query engine 大概会做：

```text
1. query planner 分析 SQL，发现 fact_sales 只需要 date_key、product_sk、quantity
2. storage layer 只读取这些 column chunks
3. 用 metadata 跳过不相关 row groups，例如 date range 不匹配的 blocks
4. 对 date_key / product_sk 做 filter 或 join
5. 得到满足条件的 row positions / bitmap
6. 用这些 row positions 去 quantity column 取值
7. 对 quantity 做 SUM
8. 输出 group by 后的 weekday/category 结果
```

这里的关键不是“完全不需要 row 概念”，而是 row 被表示成 position / bitmap，而不是一开始就 materialize 成完整 row object。

例如：

```text
product_sk = 31 bitmap:
row:  1 2 3 4 5 6
bit:  1 0 1 0 0 1

store_sk = 3 bitmap:
row:  1 2 3 4 5 6
bit:  0 0 1 1 0 1

AND result:
row:  1 2 3 4 5 6
bit:  0 0 1 0 0 1
```

结果表示 row 3 和 row 6 同时满足条件。接下来 query engine 可以只取 `quantity[3]` 和 `quantity[6]` 做 aggregation。

这也是后面 `vectorized execution` 能接上的地方：columnar data 天然是一批同类型 values，适合批量处理、SIMD、bitwise operation、cache-friendly scan。

## 写入列式存储：是不是先写 Row Store 再 Convert 成 Column Store

原书说：“所有写入首先进入面向行的、排序的内存存储。当积累了足够的写入时，它们将与磁盘上的列编码文件合并，并批量写入新文件。”

可以理解为：**是的，很多 column store 的 ingest path 会先用 row-like / in-memory buffer 接收 writes，然后批量转换成 columnar files；但这不是把整张数据库每次重新 convert 一遍。**

更准确的过程是：

```text
1. 新写入先进入内存中的 write buffer / delta store
   这里可能更接近 row-oriented representation，因为一条 insert 天然是一整行

2. buffer 积累到一定大小
   例如几万、几十万、几百万 rows

3. 系统把这一批 rows 按目标 sort key 排序
   例如按 date_key, product_sk 排序

4. 按 column 拆开
   date_key column chunk
   product_sk column chunk
   quantity column chunk

5. 对每个 column 做 encoding / compression
   dictionary encoding, bitmap encoding, RLE, delta encoding, general compression 等

6. 写成新的 immutable columnar file / segment

7. 后台可能把小文件 merge / compact 成更大的文件
```

为什么要这样？因为在 sorted column store 中，往中间插入一行很痛苦。假设表已经按 `date_key` 排序，而你插入一条很早日期的数据；如果直接原地插入，很多 column files 都要从插入点开始重写。对 compressed columns 来说更麻烦，因为插入可能改变后面很多 encoded runs。

所以 column store 通常偏向 batch writes 和 immutable files：新数据先放在小的内存/增量结构里，攒够一批后顺序写出新文件。旧文件保持不变，新文件一次写入。这也是为什么 object storage 很适合 analytical columnar storage。

查询时怎么办？查询引擎会同时看：

```text
old columnar files on disk/object storage
+ recent in-memory / delta writes
```

然后把结果合并。对分析师来说，这些细节被隐藏了，看起来数据插入后就能被查询到。

## Column Store 的 Trade-Off

Column-oriented storage 的优势主要在 analytical workloads：

```text
scan many rows
read few columns
heavy filtering / grouping / aggregation
append or batch ingest
```

它的代价也很清楚：

```text
point lookup of full row 不一定最优
frequent single-row update/delete 更麻烦
需要处理 delta store、merge、compaction、metadata
重建完整 row 需要从多个 columns 按 position 拼回来
```

所以它不是 row store 的“升级版”，而是针对不同 workload 的物理布局。

一个简单判断框架：

```text
用户点开一个订单详情，需要整行             -> row-oriented
后台交易系统频繁 update 单行状态            -> row-oriented
报表扫描一年订单，只统计 quantity/revenue   -> column-oriented
数据仓库按 date/product/category 聚合        -> column-oriented
日志/事件表 append-heavy + analytical query  -> column-oriented 很常见
```

## 总结

原书这段的主线可以压缩成一句话：**row-oriented storage 把同一行的数据放在一起，适合 OLTP；column-oriented storage 把同一列的数据放在一起，适合 OLAP，因为分析查询通常扫描大量 rows 但只访问少数 columns。**

底层上，column store 通常不是以 B-tree 作为主结构，而是使用 column chunks / row groups / immutable columnar files，加上 metadata、encoding、compression、bitmap、sort order 和 vectorized execution。

排序不会搞乱数据，因为排序发生在完整 rows 上，然后再拆成 columns；所有 columns 仍然共享同一个 row order。压缩有效，是因为同一列 values 类型相同且重复多，尤其排序后容易形成 long runs，可以用 `run-length encoding` 等方法压缩。写入方面，很多系统会先把新 rows 放入内存中的 row-like buffer，攒成 batch 后排序、拆列、压缩，写成新的 columnar files；查询引擎再把这些新写入和旧的 columnar files 统一起来。
