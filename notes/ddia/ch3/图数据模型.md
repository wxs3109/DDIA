# 图数据模型

本 note 对应 DDIA 第 3 章的“图数据模型”部分。这里的重点不是“图数据库很酷”，而是：当数据里有大量 many-to-many relationship，而且查询经常需要沿着关系走多跳时，把数据建模成 graph 会比强行塞进 relational schema 或单个 document 更自然。

图数据模型的基本单位是 `vertex` 和 `edge`。`vertex` 也叫 `node` 或 `entity`，表示一个对象；`edge` 也叫 `relationship` 或 `arc`，表示两个对象之间的关系。社交网络、网页链接、道路网络、知识图谱都可以看成 graph。

## graph 适合什么问题

关系模型能处理 many-to-many relationship，例如用 `association table` / `join table` 表示用户和组织、学生和课程、商品和订单之间的关系。但当关系类型越来越多、路径越来越长、查询越来越依赖“沿关系走几步”时，relational model 会越来越笨重。

例如，社交网络里不只是“用户认识用户”。用户、post、comment、photo、group、event、place 都可以是 `vertex`；friend、like、commented_on、member_of、checked_in_at、tagged_in、attends 都可以是 `edge`。这类数据不是一棵树，也不是少数几张固定表能舒舒服服表达的结构，而是一张不断扩展的异构 graph。

如果用 relational model 表示，通常会拆成很多 entity table 和 relationship table。例如：

```sql
CREATE TABLE users (id bigint PRIMARY KEY, name text);
CREATE TABLE posts (id bigint PRIMARY KEY, author_id bigint REFERENCES users(id), body text);
CREATE TABLE comments (id bigint PRIMARY KEY, post_id bigint REFERENCES posts(id), author_id bigint REFERENCES users(id), body text);
CREATE TABLE groups (id bigint PRIMARY KEY, name text);
CREATE TABLE events (id bigint PRIMARY KEY, name text, group_id bigint REFERENCES groups(id), place_id bigint REFERENCES places(id));
CREATE TABLE places (id bigint PRIMARY KEY, name text);

CREATE TABLE friendships (user_id bigint REFERENCES users(id), friend_id bigint REFERENCES users(id));
CREATE TABLE post_likes (user_id bigint REFERENCES users(id), post_id bigint REFERENCES posts(id));
CREATE TABLE group_memberships (user_id bigint REFERENCES users(id), group_id bigint REFERENCES groups(id));
CREATE TABLE event_attendees (user_id bigint REFERENCES users(id), event_id bigint REFERENCES events(id));
CREATE TABLE checkins (user_id bigint REFERENCES users(id), place_id bigint REFERENCES places(id), checked_in_at timestamptz);
```

这本身还不算问题，relational database 很擅长管理这些表。笨重出现在查询需要沿多种 relationship 走路径时。比如你想做一个推荐：“找出 Wenbo 的朋友参加过的、发生在 Wenbo 最近 check in 过的同一个 place 的 event，并且这些 event 所属 group 里也有 Wenbo 的朋友。”这个查询会把 `friendships`、`event_attendees`、`events`、`places`、`checkins`、`group_memberships` 等表串起来：

```sql
SELECT DISTINCT e.id, e.name
FROM users me
JOIN friendships f1 ON f1.user_id = me.id
JOIN event_attendees ea ON ea.user_id = f1.friend_id
JOIN events e ON e.id = ea.event_id
JOIN checkins c ON c.user_id = me.id AND c.place_id = e.place_id
JOIN group_memberships gm_event ON gm_event.group_id = e.group_id
JOIN friendships f2 ON f2.user_id = me.id AND f2.friend_id = gm_event.user_id
WHERE me.name = 'Wenbo';
```

这里每多一种 relationship，就要多一张 relationship table 和一段 join 逻辑；如果路径长度不固定，例如“朋友的朋友的朋友参加的 event”，SQL 还要引入 recursive CTE。graph model 不是说这些关系数据库不能表示，而是把它们拆在很多表和 join 里之后，query 的形状不再直观；你脑子里想的是“沿 friend -> attends -> place -> group member 走一圈”，SQL 里看到的是一串 join。

graph 的查询也经常是 traversal：

- 查共同好友：比较两个人的 friend edge 集合。
- 推荐好友：从朋友、同城、同校、同 group 等路径扩展。
- 权限判断：某个用户是否通过 friend / group / organization 等关系可达。
- 地理层级：一个 city `WITHIN` 一个 state，state `WITHIN` country，country `WITHIN` continent。

这类查询的核心不是“查某一列等于某个值”，而是“沿着某些 relationship 找到满足条件的路径”。

> [!NOTE] Wenbo 注
> graph database 的优势通常不是存储一个单独的关系，而是处理“关系之间的组合”。如果只是 `user_id -> org_id` 这种简单 many-to-many，relational join table 很好。但如果你经常问“从这个人出发，经过 friend、member_of、located_in、works_at 这些关系，能不能找到某类对象”，graph model 的表达会自然很多。

## 业界例子：Facebook social graph / TAO

书里提到 Facebook，不是在说 Facebook 直接用了 Neo4j、Cypher、SPARQL 或 Datalog。它主要是在说明一种典型的 graph-shaped application：社交产品里的对象类型和关系类型非常多，而且很多核心功能都依赖 traversal。

Facebook 公开论文里介绍过 `TAO`，全称是 `Facebook's Distributed Data Store for the Social Graph`。TAO 是 Facebook 为 social graph 定制的 distributed data store / caching layer。它的抽象不是本章后面讲的标准 `property graph database` 查询语言，而是更接近内部 API：应用通过 TAO 读写 `object` 和 `association`。

可以把 TAO 的概念和本章术语对应起来：

| TAO / Facebook 术语 | 本章 graph 术语 | 例子 |
| --- | --- | --- |
| `object` | `vertex` | user、page、post、comment、photo、event、place |
| `association` | `edge` | friend、like、commented_on、member_of、tagged_in |
| association type | edge `label` | `FRIEND_OF`、`LIKES`、`AUTHORED` |
| object fields | vertex `properties` | name、timestamp、privacy、title |

例如，在 graph 的思维里，可以把 Facebook 的数据想成：

```text
(user:Wenbo) -[:FRIEND_OF]-> (user:Alice)
(user:Alice) -[:LIKES]-> (post:9001)
(post:9001) -[:AUTHORED_BY]-> (user:Bob)
(user:Wenbo) -[:MEMBER_OF]-> (group:DistributedSystems)
(post:9001) -[:POSTED_IN]-> (group:DistributedSystems)
```

这类数据可以支撑很多产品功能：

- News Feed：哪些 post 和你有关，取决于 friend、follow、group membership、like、comment 等 association。
- 权限判断：某个 viewer 是否能看到某条 content，可能依赖 friend、group member、blocked、privacy setting 等关系。
- 推荐：推荐好友、page、group、event，通常都要沿着 social graph 做多跳扩展和排序。
- 计数和摘要：某个对象有多少 like、comment、share，哪些朋友也参与过。

TAO 的底层不是“单机图数据库”。根据 TAO 论文，它面向超大规模读负载，底层持久化使用 sharded MySQL，上层有 TAO cache、分片、复制、跨数据中心部署等机制。粗略可以理解为：

```text
Application
    -> TAO API: object / association reads and writes
    -> distributed cache and routing
    -> sharded MySQL persistent storage
```

所以 Facebook 这个例子的重点是：它的业务数据天然是 graph；但在工程实现上，它没有直接套用本章里的 `Cypher` / `SPARQL` / `Datalog`，而是为自己的访问模式做了专用 distributed graph store。

参考文献：

- Nathan Bronson et al., [TAO: Facebook's Distributed Data Store for the Social Graph](https://www.usenix.org/conference/atc13/technical-sessions/presentation/bronson), USENIX ATC 2013.

## 业界例子：Google / search engine Knowledge Graph

另一个非常成功的 graph-shaped system 是搜索引擎里的 `Knowledge Graph`。这也正好对应书中提到的知识图：搜索引擎会记录常见实体（person、organization、place、movie、book、product 等）以及实体之间的事实关系。

例如，传统全文检索看到的是字符串：

```text
"Barack Obama"、"Honolulu"、"Michelle Obama"、"United States"
```

Knowledge Graph 更关心这些字符串背后的 entity 和 relationship：

```text
(Barack Obama) -[:BORN_IN]-> (Honolulu)
(Honolulu) -[:LOCATED_IN]-> (Hawaii)
(Hawaii) -[:PART_OF]-> (United States)
(Barack Obama) -[:SPOUSE]-> (Michelle Obama)
(Barack Obama) -[:HELD_POSITION]-> (President of the United States)
```

这样搜索系统就不只是匹配关键词，而是能理解“things, not strings”。例如用户搜 `Obama wife`，系统可以把 `Obama` 解析成 Barack Obama 这个 entity，把 `wife` 映射到 spouse relationship，然后返回 Michelle Obama。用户搜 `where was Obama born`，系统可以沿 `BORN_IN` relationship 找到 Honolulu，并进一步知道 Honolulu 在 Hawaii、Hawaii 在 United States。

这和本章的 graph model 很像：

- entity 是 `vertex`。
- relationship / fact 是 `edge` 或 triple 的 `predicate`。
- entity 的属性，比如出生日期、职业、别名，可以是 properties 或 literal object。
- 查询常常是 graph pattern matching 或 traversal，而不只是 keyword matching。

Knowledge Graph 的业界价值很大：它能支持搜索结果右侧的知识面板、实体消歧、问答、推荐、语义搜索，以及把来自网页、Wikidata、结构化数据、企业数据库的信息合并到同一张 entity graph 中。

参考文献：

- Google, [Introducing the Knowledge Graph: things, not strings](https://blog.google/products/search/introducing-knowledge-graph-things-not/), 2012.
- Natasha Noy et al., [Industry-Scale Knowledge Graphs: Lessons and Challenges](https://cacm.acm.org/magazines/2019/8/238342-industry-scale-knowledge-graphs/fulltext), Communications of the ACM, 2019.

## graph 的两种表示方式

图可以用不同方式在底层表示。书中提到两个基础概念：`adjacency list` 和 `adjacency matrix`。

`adjacency list` 是给每个 vertex 存它相邻 vertex 的 ID。它适合 `graph traversal`，因为从一个 vertex 出发，能快速找到它的邻居，再从邻居继续扩展。

`adjacency matrix` 是一个二维矩阵，行和列都是 vertex。如果两个 vertex 之间有 edge，对应位置是 1；没有 edge，就是 0。它适合某些机器学习、线性代数或图算法场景，但对 sparse graph 很浪费。

现实中的社交 graph 通常是 very sparse。理论上任意两个用户之间都可能有关系，但每个用户实际只连接极少数其他用户、post、group、event。因此实际存在的 edge 远远少于所有可能的 vertex pair。对这种数据，用 adjacency list / edge table 只存真实存在的 edge 更合理。

## property graph

`property graph` 也叫 `labeled property graph`，是 Neo4j、Memgraph、KuzuDB 等系统使用的模型。它把 graph 看成带 label 和 properties 的 vertex / edge。

每个 `vertex` 包含：

- 唯一标识符。
- `label`，表示对象类型，例如 `Person`、`Location`、`Event`。
- 一组出边。
- 一组入边。
- `properties`，也就是 key-value 属性集合。

每条 `edge` 包含：

- 唯一标识符。
- `tail vertex`，也就是 edge 开始的 vertex。
- `head vertex`，也就是 edge 指向的 vertex。
- `label`，表示 relationship 类型，例如 `BORN_IN`、`LIVES_IN`、`WITHIN`。
- `properties`，也就是 key-value 属性集合。

从 relational 角度看，property graph 可以用两张表模拟：`vertices` 表和 `edges` 表。`edges` 表里保存 `tail_vertex`、`head_vertex`、`label` 和 `properties`。为了快速 traversal，通常需要在 `tail_vertex` 和 `head_vertex` 上建 index，这样才能高效找一个 vertex 的出边和入边。

property graph 的重要特点是：

1. 任意 vertex 可以通过 edge 连到任意其他 vertex。
2. 给定一个 vertex，可以高效找到它的 incoming edge 和 outgoing edge。
3. 用不同的 label 区分不同类型的 vertex 和 edge，可以在同一张 graph 里存很多不同类型的数据。

这让 property graph 很适合 schema evolvability。新增一种对象或关系时，通常可以新增 label 和 properties，而不必重构大量固定表结构。

## graph model 的限制：binary relationship

普通 property graph 的 `edge` 通常只能连接两个 vertex，也就是表达 `binary relationship`。例如：

```text
Lucy -[:LIVES_IN]-> London
```

这条 edge 表达的是 Lucy 和 London 之间的关系。

但现实中有些事实是 `ternary relationship` 或更高阶的关系。例如：

- “Alice 在 Store A 购买 Product B”。
- “Wenbo 在 Microsoft 担任 engineer”。
- “Student S 在 Semester T 选了 Course C，由 Teacher P 教”。

这些关系不是两个对象之间的简单连接，而是多个 entity 共同构成一个事实。

在 relational model 里，这很自然：用一张 `purchase`、`employment`、`enrollment` 这样的 join table，一行里可以有多个 foreign key。

在普通 graph 里，常见做法是把 relationship 本身也 reify 成一个 vertex。例如创建一个 `Employment` vertex，再用 edge 连接 person、organization、role、time 等对象。这会让模型多一层，但可以表达 higher-order relationship。

`hypergraph` 是另一种思路。它允许一条 `hyperedge` 同时连接多个 vertex，因此更适合直接表达多方关系。不过很多主流 graph database 使用的是 property graph，而不是 hypergraph。

## Cypher

`Cypher` 是 property graph 的查询语言，最初由 Neo4j 创建，后来发展为 `openCypher`。它的特点是用图形化的 pattern syntax 表达 traversal。

例如：

```cypher
MATCH
    (person) -[:BORN_IN]-> () -[:WITHIN*0..]-> (:Location {name:'United States'}),
    (person) -[:LIVES_IN]-> () -[:WITHIN*0..]-> (:Location {name:'Europe'})
RETURN person.name
```

这个查询的意思是：找出同时满足两个条件的 `person`：

1. `person` 有 `BORN_IN` edge 指向某个位置，并且这个位置通过零次或多次 `WITHIN` edge 属于 `United States`。
2. 同一个 `person` 有 `LIVES_IN` edge 指向某个位置，并且这个位置通过零次或多次 `WITHIN` edge 属于 `Europe`。

`WITHIN*0..` 是 variable-length traversal，意思是沿 `WITHIN` edge 走 0 次或多次。走 0 次表示当前 vertex 自己就匹配；走多次表示可以从 city 到 state 到 country 到 continent 一路爬层级。

Cypher 是 `declarative query language`。你描述的是要匹配的 graph pattern，而不是规定数据库必须用 BFS、DFS 还是其他算法。query planner 可以决定从哪里开始，例如先用 index 找到 `United States` 和 `Europe` 这两个 anchor vertex，再沿反方向的 `WITHIN` edge 展开，最后做 join / intersection。

> [!NOTE] Wenbo 注
> Cypher 的漂亮之处是，它让“路径形状”直接出现在查询里。你看到 `(person)-[:BORN_IN]->()-[:WITHIN*0..]->(:Location {name:'United States'})`，就能直观看出 person 的出生地要一路属于 United States。SQL 也能做，但表达会绕很多。

## SQL 中的 graph query

graph 数据可以存在 relational database 里，例如用 `vertices` 和 `edges` 两张表。但 graph query 在 SQL 中会很笨重，因为 traversal 本质上是不断 join `edges` 表。

普通 SQL join 的数量通常是固定的：你写几个 join，数据库就执行几个 join。但 graph traversal 的路径长度可能不固定，例如 `WITHIN*0..` 可能走 0 步、1 步、2 步或更多步。这个时候需要 `recursive common table expression`，也就是 `WITH RECURSIVE`。

书中的 SQL 示例用递归 CTE 构造了几个集合：

- `in_usa`：United States 境内所有 location 的 vertex ID。
- `in_europe`：Europe 境内所有 location 的 vertex ID。
- `born_in_usa`：出生地在 `in_usa` 里的 person。
- `lives_in_europe`：居住地在 `in_europe` 里的 person。

最后 join `born_in_usa` 和 `lives_in_europe`，得到同时满足两个条件的人。

这个例子说明：一个 4 行 Cypher 查询，在 SQL 里可能需要几十行 recursive CTE。不是 SQL 做不到，而是数据模型和查询语言不贴合问题形状。

SQL 对 graph query 的支持也在发展，例如 GQL / SQL/PGQ 方向试图把 graph pattern matching 加到 SQL 标准附近。

## triple store

`triple store` 和 property graph 很像，只是术语不同。triple store 把所有信息都表示成三元组：

```text
(subject, predicate, object)
```

例如：

```text
(Jim, likes, bananas)
```

其中：

- `subject` 通常对应 graph 里的 vertex。
- `predicate` 表示属性名或 edge label。
- `object` 可以是 literal value，也可以是另一个 vertex。

如果 object 是原始值，那么 triple 表达的是属性：

```text
(lucy, birthYear, 1989)
```

这类似于 property graph 里的：

```json
{"birthYear": 1989}
```

如果 object 是另一个 vertex，那么 triple 表达的是 edge：

```text
(lucy, marriedTo, alain)
```

这类似于：

```text
lucy -[:marriedTo]-> alain
```

一些数据库实际存的不是严格 3-tuple，而是 4-tuple 或 5-tuple，例如加上 graph ID、transaction ID、删除标记等 metadata。但只要核心仍是 subject-predicate-object 结构，书里仍然把它们归为 triple store。

## Turtle / Notation3

`Turtle` 是 RDF 数据的一种文本编码格式，是 `Notation3` 的子集。它可以把 triple store 数据写成比较紧凑的文本。

完整写法会重复 subject：

```turtle
_:lucy a :Person.
_:lucy :name "Lucy".
_:lucy :bornIn _:idaho.
```

Turtle 也支持用分号把同一个 subject 的多个 predicate-object 写在一起：

```turtle
_:lucy a :Person; :name "Lucy"; :bornIn _:idaho.
```

这里的 `_:lucy` 是 blank node 名称，只在当前文件里有意义，用来说明多条 triple 引用的是同一个 vertex。`a` 是 RDF 里的快捷写法，表示 `rdf:type`，也就是“这个资源是什么类型”。

## RDF

`RDF` 是 `Resource Description Framework`，是语义网背景下提出的数据模型。Turtle 只是 RDF 的一种编码方式；RDF 也可以用 XML 表达，也就是 `RDF/XML`。

RDF 的一个重要特点是：subject、predicate、object 通常用 URI 标识。这样做是为了互联网范围的数据交换，避免不同组织对同一个词使用不同含义时发生冲突。

例如，一个系统里的 `within` 可以是：

```text
http://my-company.com/namespace#within
```

另一个系统里的 `within` 可以是：

```text
http://other.org/foo#within
```

它们表面上都叫 `within`，但 URI 不同，因此语义上不会混淆。

在书中的例子里，用 `urn:example:` 作为 prefix，这样就不用每次都写完整 URI。

> [!NOTE] Wenbo 注
> RDF 的 URI 设计有点像给 predicate 加 namespace。它解决的是“跨组织共享数据时，怎么知道大家说的是不是同一个概念”。在公司内部系统里你可能觉得这很啰嗦，但在知识图谱、开放数据、语义网场景里很重要。

## SPARQL

`SPARQL` 是 RDF / triple store 的查询语言。它比 Cypher 更早，Cypher 的 graph pattern matching 很大程度上借鉴了 SPARQL。

书中的查询是：

```sparql
PREFIX : <urn:example:>

SELECT ?personName WHERE {
 ?person :name ?personName.
 ?person :bornIn / :within* / :name "United States".
 ?person :livesIn / :within* / :name "Europe".
}
```

它和 Cypher 查询表达的是同一个问题：找出生在 United States、居住在 Europe 的人。

SPARQL 里变量以 `?` 开头，例如 `?person`、`?personName`。路径表达式：

```sparql
?person :bornIn / :within* ?location.
```

大致对应 Cypher：

```cypher
(person) -[:BORN_IN]-> () -[:WITHIN*0..]-> (location)
```

RDF 不区分 property 和 edge；一切都是 predicate。因此匹配属性和匹配关系使用同一种语法。例如：

```sparql
?usa :name "United States".
```

表示找到一个 subject，它有 predicate `:name`，object 是字符串 `"United States"`。

## Datalog

`Datalog` 是一种更老的逻辑查询语言，源自 1980 年代。它基于 relational model，而不是直接基于 graph model，但特别擅长 recursive query，因此适合表达 graph traversal。

Datalog 数据库由 `fact` 组成，每个 fact 类似 relational table 的一行：

```datalog
location(2, "United States", "country").
within(3, 2).
person(100, "Lucy").
born_in(100, 3).
```

Datalog 查询通过 `rule` 派生新的虚拟表。规则的形式是：

```datalog
head :- body.
```

可以理解为：如果 `body` 里的条件都成立，那么推出 `head` 这个事实。

书中的核心递归规则是：

```datalog
within_recursive(LocID, PlaceName) :- location(LocID, PlaceName, _).

within_recursive(LocID, PlaceName) :- within(LocID, ViaID),
 within_recursive(ViaID, PlaceName).
```

第一条规则说：一个 location 在它自己里面。第二条规则说：如果 `LocID` within `ViaID`，而 `ViaID` 又在某个 `PlaceName` 里面，那么 `LocID` 也在这个 `PlaceName` 里面。

这就是递归 traversal。通过不断应用规则，系统可以推出 Idaho 在 United States 内，也在 North America 内。

Datalog 的思维方式和 Cypher / SPARQL 不同。Cypher / SPARQL 是直接写一个 graph pattern；Datalog 是一步一步定义 derived relation，就像把复杂逻辑拆成多个函数。

> [!NOTE] Wenbo 注
> Datalog 最值得注意的是 rule 可以引用自己，所以天然适合表达 transitive closure，例如“某地是否属于某个更大的地理区域”“某人是否通过组织层级归属于某部门”“依赖图里 A 是否间接依赖 B”。这类问题在 SQL 里通常要用 recursive CTE。

## GraphQL

`GraphQL` 名字里有 graph，但它不是本章前面那种 graph database query language。它主要是 API query language，用来让客户端请求一个特定形状的 JSON response。

GraphQL 的典型目标是：移动端或前端页面需要哪些字段，就在 query 里写哪些字段，服务器返回同样结构的 JSON。

例如聊天应用可以查询：

- 用户有权访问的 channels。
- 每个 channel 的最近 50 条 messages。
- 每条 message 的 timestamp、content。
- sender 的 fullName 和 imageUrl。
- 如果是 reply，再返回被回复消息的一些字段。

GraphQL 的 response 通常会重复嵌入数据。例如同一个用户发了多条消息，sender 信息可能在每条 message 里重复。这是为了让客户端渲染简单，避免客户端再到处追 ID 做 join。

GraphQL 和 graph database 的关键区别：

- GraphQL 可以跑在 relational database、document database、graph database 或微服务 API 之上。
- GraphQL query 只能请求 schema 明确暴露的字段和关系。
- GraphQL 不允许任意 recursive query，因为客户端 query 来自不受信任来源，必须控制成本。
- GraphQL 不适合表达“找所有在美国出生且住在欧洲的人”这种任意搜索，除非服务端专门暴露了这个 search capability。

所以 GraphQL 更像 client-server API 层的数据选择语言，而不是底层 database query language。

## 几种模型的对比

| 模型 / 语言 | 核心用途 | 适合表达 | 主要特点 |
| --- | --- | --- | --- |
| `property graph` | graph database data model | vertex / edge / properties | 适合 heterogeneous graph 和 traversal |
| `Cypher` | property graph query language | graph pattern matching | 语法直观，适合 variable-length traversal |
| SQL recursive CTE | relational query language 的递归扩展 | 在 relational table 中模拟 traversal | 能做，但语法笨重 |
| `triple store` | subject-predicate-object 数据模型 | RDF / knowledge graph | property 和 edge 都是 predicate |
| `RDF` | Web-scale data exchange model | URI 标识的三元组 | 重视 namespace 和语义互操作 |
| `SPARQL` | RDF query language | triple pattern / path expression | 和 Cypher 很像，但基于 triple |
| `Datalog` | recursive relational query language | rule-based derived relation | 适合递归、transitive closure、复杂逻辑 |
| `GraphQL` | API query language | 客户端请求 JSON shape | 不是 graph database 查询语言 |

## 本节的核心 takeaway

当数据主要是一对多树状结构、通常整体读取时，document model 很自然。当数据有大量 many-to-many relationship，并且查询常常需要沿 relationship 走多跳时，graph model 更自然。

graph model 并不神秘。它的核心是把对象表示为 `vertex`，把关系表示为 `edge`，并让 traversal 成为一等操作。property graph、triple store、RDF 都是在表达这个基本思想，只是术语、约束和生态不同。

Cypher 和 SPARQL 的优势是直接表达 graph pattern；Datalog 的优势是用 rule 表达递归逻辑；SQL 也能模拟 graph query，但在 variable-length traversal 上往往更冗长。GraphQL 虽然名字里有 graph，但主要是 API 层工具，不应和 graph database 混为一谈。
