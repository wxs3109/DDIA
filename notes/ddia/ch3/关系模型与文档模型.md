# 关系模型与文档模型

这一节不是在给 `relational database` 和 `document database` 排名，而是在建立一个判断框架：数据之间的关系是什么形状，哪些数据经常一起读取，哪些信息会频繁变化，哪些查询必须快，哪些一致性成本可以接受。`relational model` 和 `document model` 都能表达很多相同的事实，但它们让不同的结构变得自然或别扭。

## relational model 的位置

`relational model` 把数据组织成 `relation`，在 SQL 里通常表现为 `table`。每个 `table` 由很多 `row` 组成，每个 `row` 有一组 `column`。不同 `table` 之间通过 ID 建立引用，再通过 `join` 把相关信息组合回来。

这个模型的核心优势不是“数据长得像表格”，而是它非常适合表达共享实体、引用关系、约束和组合查询。比如一个 `user` 引用一个 `region`，很多 `user` 可以引用同一个 `region`；一个 `person` 可以属于多个 `organization`，一个 `organization` 也可以有很多 `person`。这些关系在 `relational model` 中通常都能用 `foreign key`、`join table`、`secondary index` 和 `join` 表达清楚。

> [!NOTE] Wenbo 注
> `foreign key` 是 table 之间表达引用关系的字段。例如 `users.region_id` 指向 `regions.id`，意思是这个 user 属于哪个 region。它的重点是“关系和约束”：数据库可以检查 `users.region_id` 引用的 region 是否真的存在，避免出现指向不存在记录的脏数据。
>
> `secondary index` 是为了让非 primary key 字段也能被快速查询的数据结构。比如 `positions` table 里 primary key 可能是 `position_id`，但我们经常要按 `user_id` 查某个人的所有 positions，或者按 `org_id` 查某个 organization 下的所有人。给 `user_id`、`org_id` 建 `secondary index` 后，数据库就不用全表扫描。
>
> 一个 table 可以有很多个 `secondary index`。例如 `users` table 可以对 `email` 建 index 用于登录查找，对 `region_id` 建 index 用于按地区筛选，对 `created_at` 建 index 用于按注册时间排序。每个 index 都是在给某一种查询路径开“快捷通道”。
>
> 但 index 不是免费的。它用额外成本换查询速度：
>
> * 更多 storage：每个 index 都要额外保存一份数据结构。
> * 更慢 write：insert、update、delete 一行时，不只要改 table 本身，还要同步更新相关 indexes。
> * 更复杂 planning：index 多了以后，query optimizer 要选择用哪个 index；建错 index 可能没用，还会拖慢写入。
> * 更新热点：如果某个 indexed field 经常变化，对应 index 也要频繁维护。
>
> 所以 `secondary index` 的本质 trade-off 是：牺牲写入速度、存储空间和维护成本，换取某些读取查询更快。
>
> 简单说：`foreign key` 主要表达“这两张表有什么关系”，`secondary index` 主要解决“按这个字段查得快不快”。它们经常一起出现，但不是一回事。

`relational model` 长期占主导，是因为很多业务数据都需要稳定的结构、约束、transaction 和复杂查询。后来的 `NoSQL` 和 `NewSQL` 并不是简单取代 SQL，而是把注意力放到其他诉求上：更灵活的 `schema flexibility`、更容易横向扩展、开源生态，以及更贴近某些应用对象结构的数据模型。`document model` 就是在这个背景下流行起来的，它通常用 JSON-like document 表示数据。

## object-relational impedance mismatch 与 ORM

应用代码里的数据通常以 `object` 或 `data structure` 的形式存在，但 `relational database` 里是 `table`、`row`、`column`。这两种形状不同，就会产生 `object-relational impedance mismatch`。

例如应用里可能希望使用这样的对象：

```text
User
  id
  name
  region
  positions
  education
  contactInfo
```

但在 `relational schema` 里，为了表达 one-to-many relationship，通常会拆成多个 `table`：

```text
users
regions
positions
education
contact_info
```

`ORM`，也就是 `object-relational mapping`，就是在 application object 和 relational schema 之间做映射。它把 `class` 映射到 `table`，把 object attribute 映射到 `column`，把 object relationship 映射到 `foreign key` 或 `join table`。

Python 里常见的 `ORM` 有 `SQLAlchemy ORM` 和 `Django ORM`。`SQLAlchemy` 可以用比较底层的 SQL expression，也可以用 ORM class 映射 table；`Django ORM` 则和 Django 的 `Model` 体系绑定得更紧。

.NET / C# 里最主流的 full ORM 是 `Entity Framework Core`，也就是 `EF Core`。它通过 C# class 定义 entity，用 `DbContext` 管理查询和变更追踪，再生成 SQL。`Dapper` 也很常见，但它更像 `micro-ORM` 或 lightweight SQL mapper：它主要让你手写 SQL，然后把 SQL result 映射成 C# object；它不太负责复杂的 `change tracking`、`object graph tracking`、`migration` 等完整 ORM 能力。所以在 C# 里，`EF Core` 更接近 full ORM，`Dapper` 更接近 micro-ORM。

`ORM` 的价值是减少重复的映射代码，让简单 CRUD 更省事；但它不能让 `relational model` 和 object model 的差异消失。特别是复杂查询、`N+1 query problem`、`join` 性能、`schema design`，仍然需要理解数据库本身。比如一个页面要显示 N 条评论以及每条评论的作者，如果 ORM 为每条评论单独查一次作者，就会变成 N+1 次 query；手写 SQL 可能一个 `join` 就完成了。

## document model 与 tree-like data

`document model` 通常把一组相关数据存在一个 JSON-like document 中。它常被认为更接近 application object，因为很多 object 本身就是嵌套结构。

比如一个 LinkedIn profile 可以写成一个 document：用户的基本字段在顶层，`positions`、`education`、`contact_info` 作为嵌套结构放进去。这个结构背后其实是一棵 tree：一个 user 有多个 positions，有多个 education entries，有一个 contact info。这里的关系是从一个 root document 向下展开的 one-to-many，或者更准确说，在很多 profile 场景里是 one-to-few。

这也是 `document model` 最自然的地方：如果数据是 tree-like，而且经常整体读取，那么把它放在一个 document 里会很舒服。读取 profile 时不用把 `users`、`positions`、`education`、`contact_info` 多个 table join 回来；一个 document 就能拿到大部分页面需要的数据。

但 `document` 和 one-to-many 不是一回事。one-to-many 是一种 relationship，`document model` 只是表达它的一种方式。one-to-many 也可以用 `relational model` 表达：父表一行，子表多行，子表用 `foreign key` 指向父表。

`document model` 适合的是“父子数据经常一起读取、子项数量可控、子项很少被外部单独引用”的 one-to-many。如果子项数量可能非常大，比如热门帖子下的几百万条 comments，把所有 comments 嵌进一个 document 就会很糟。如果嵌套项需要被别的地方精确引用，`document model` 也会别扭，因为你通常不能像引用一行 table 那样自然地引用 document 内部数组里的某个 item。

所以 MongoDB 这种 JSON-like model 并不是“比 relational 更适合 object”的通用答案。它确实能减少某些 `object-relational impedance mismatch`，但一旦数据里出现大量 cross-reference、many-to-many relationship、复杂查询，`relational model` 通常更清晰。

## normalization、denormalization 与 join

`normalization` 的核心是：一个事实只存一份，其他地方保存引用。引用通常是 ID。

`denormalization` 的核心是：为了读取更快或更方便，把某些事实复制到多个地方，或者把某个昂贵查询的结果预先保存下来。

例如 `region` 信息可以这样 normalized：

```text
users
  id = 251
  name = Barack Obama
  region_id = us:91

regions
  id = us:91
  region_name = Washington, DC, United States
```

读取用户并显示 region name 时，需要通过 ID 做 `join`：

```sql
SELECT users.*, regions.region_name
FROM users
JOIN regions ON users.region_id = regions.id
WHERE users.id = 251;
```

如果 denormalized，就可以把人类可读的 region name 直接复制进 user record：

```text
users
  id = 251
  name = Barack Obama
  region_name = Washington, DC, United States
```

这样读取时不需要 `join`，但代价是 `region_name` 被复制到了很多地方。如果将来城市改名、行政区划变化、翻译更新，所有副本都要更新。只要有一部分副本没更新，就会产生 inconsistency。

因此可以先记住一个粗略判断：

* `normalized data` 通常写入和维护更稳，因为事实只有一个权威位置。
* `denormalized data` 通常读取更快，因为少做 `join` 或少做现场计算。
* `normalized data` 的查询可能更慢，因为需要通过 ID 做 `join`。
* `denormalized data` 的写入和一致性维护更贵，因为有多个副本。

但 `normalization` 不是“只要用了 ID 就一定 normalized”，`denormalization` 也不是“必须复制完整内容”。更准确地说，判断点是：某个事实或某个查询结果是否被保存了多份。

## social network timeline 中的局部 denormalization

social network timeline 的例子容易误解，因为 X / Twitter 的 `materialized timeline` 里仍然存的是 `post ID` 和 `user ID`，看起来好像还是 normalized，而且读取时还要 `hydrating`，还要查别的 table 或 service。

这里的关键是：它 denormalized 的不是 post text、like count 或 profile image，而是 timeline membership，也就是“哪些 post 应该出现在某个用户 timeline 里”这个昂贵查询结果。

如果完全 normalized，每次用户打开 timeline，都可以现场做类似这样的查询：

```sql
SELECT posts.id, posts.sender_id
FROM posts
JOIN follows ON posts.sender_id = follows.followee_id
WHERE follows.follower_id = current_user
ORDER BY posts.timestamp DESC
LIMIT 1000;
```

这表示：先找出当前用户 follow 的人，再找这些人的 posts，再排序。对于超大规模 social network，这个 `posts JOIN follows` 在 timeline 高频读取场景下会非常贵。

`materialized timeline` 的做法是：当某个用户发 post 时，系统提前把这个 post ID 写入其 followers 的 timeline。也就是说，原本读取时才计算的 `posts JOIN follows` 结果，被提前计算并保存了。

> [!NOTE] Wenbo 注
> 对，这一步就是 `fanout`：一个用户发出一条 post，系统把这条 post 分发到很多 followers 的 timeline 里。这里的 `fanout` 不是另一种 data model，而是维护 `materialized timeline` 的写入过程。
>
> 换句话说，`materialized timeline` 是被保存下来的 denormalized read model；`fanout` 是把新 post 写进这些 read model 的动作。

```text
normalized 方式：读取 timeline 时现场计算 posts JOIN follows
denormalized 方式：写入 post 时提前把结果写进 followers 的 materialized timeline
```

这就是 denormalization：保存了一个本来可以通过 normalized data 计算出来的派生结果。

它仍然只保存 ID，是因为不同信息的变化频率不同。timeline membership 相对适合预计算；但 `like count`、`reply count`、`username`、`profile image` 变化很快，如果也复制到每个 follower 的 timeline，更新成本会非常高。因此系统读取 timeline 时还会做 `hydrating`。

`hydrating` 可以理解成“把瘦 ID 补全成完整对象”。timeline 里先保存轻量的 `post ID`、`user ID`；真正展示时，再根据这些 ID 查 `post content`、`like count`、`reply count`、`username`、`profile image` 等信息，组装成前端可以展示的完整 timeline item。

这个例子说明，真实系统经常是混合的：最贵、最稳定、最值得缓存的部分被 denormalized；变化快、复制成本高的部分仍保持 normalized，然后在读取阶段通过 join 或 hydrating 补全。

## 多对一与多对多关系

`region_id` 是 many-to-one relationship：很多 users 可以属于同一个 region。`positions` 和 `education` 在 profile 里是 one-to-many 或 one-to-few：一个 user 有多个 positions，但每个 position 通常属于一个 user。

如果把 `organization` 和 `school` 也抽成独立 entity，那么关系会变成 many-to-many：一个 person 可以在多个 organization 工作，一个 organization 也可以有很多 person。

在 `relational model` 中，many-to-many relationship 通常用 `join table`，也叫 `association table`：

```text
users
  id
  name

organizations
  id
  name

positions
  user_id
  org_id
  job_title
  start
  end
```

这里 `positions` 不只是“中间表”。它本身承载了关系上的属性，比如 `job_title`、`start`、`end`。查某个人工作过的 organization，可以从 `positions.user_id` 出发 join 到 `organizations`；查某个 organization 下有哪些 person，可以从 `positions.org_id` 出发 join 到 `users`。为了让两个方向都快，通常会在 `positions.user_id` 和 `positions.org_id` 上建 `secondary index`。

在 `document model` 中，many-to-many 没有这么自然。可以在 user document 里保存 `org_id`：

```json
{
  "user_id": 251,
  "positions": [
    {"job_title": "President", "org_id": 513},
    {"job_title": "US Senator", "org_id": 514}
  ]
}
```

这样从 person 查 organization 很方便。但如果要从 organization 反查所有 person，就需要对嵌套字段 `positions.org_id` 建 `secondary index`，否则就要扫描大量 user documents。

也可以在两边都存 ID：user document 存 `org_id list`，organization document 存 `user_id list`。这样两个方向读取都方便，但同一段 relationship 被保存了两份，因此这是 denormalized relationship。它的风险是两边可能不一致。

因此，本节的一个重要判断是：tree-like one-to-few data 适合 document；many-to-one 和 many-to-many 更适合 normalized representation，尤其当你需要双向查询、约束和一致性时。

## analytics 中的 star schema、snowflake schema 与 OBT

`star schema` 和 `snowflake schema` 仍然属于 `relational model`。它们没有离开 table、row、foreign key、join，只是它们服务的是 `analytics` / `data warehouse`，不是普通 `OLTP` application。

`OLTP` 关心的是高频、小规模、面向业务操作的读写，比如注册用户、创建订单、更新资料。`analytics` 关心的是大量历史事件上的过滤、聚合、分组和比较，比如每个月销售额、不同地区表现、不同 product category 的利润率。

在 `star schema` 中，中心是 `fact table`。`fact table` 的每一行通常是一条事件，例如一次销售、一次 click、一次 page view。围绕它的是多个 `dimension table`，描述这条事件的 who、what、where、when、how、why。

```text
              dim_product
                   |
dim_customer -- fact_sales -- dim_store
                   |
                dim_date
```

`fact_sales` 里会有销售价格、成本、数量等度量，也会有指向 `dim_product`、`dim_store`、`dim_customer`、`dim_date` 的 foreign key。分析查询通常从巨大的 `fact table` 出发，join 多个 `dimension table`，再 group by、filter、aggregate。

`snowflake schema` 是把某些 `dimension table` 继续 normalized。例如 `dim_product` 不直接保存 brand 和 category 的全部信息，而是引用 `dim_brand`、`dim_category`：

```text
fact_sales -> dim_product -> dim_brand
                         -> dim_category
```

所以 `snowflake schema` 比 `star schema` 更 normalized；但 `star schema` 通常对分析师更简单，因为维度信息更集中，少一些 join。

`one big table`，也就是 `OBT`，则走向更强的 denormalization：把 dimension table 的信息提前 join 好，直接折叠进一个很宽的 fact-like table。这样查询可能更快、更简单，但存储更多重复数据。

这里之所以看起来和前文割裂，是因为讨论对象从 application data model 切换到了 data warehouse modeling pattern。它们仍然是 relational，只是优化目标变了：前面更关注 OLTP 的数据建模和关系表达；这里更关注 analytics 的批量读取、聚合和分析便利性。

在 analytics 场景里，denormalization 的代价通常更容易接受，因为数据多是历史事件日志，更新较少，主要是读和聚合。OLTP 中 denormalization 带来的 consistency 和 write amplification 问题，在 data warehouse 里通常没那么尖锐。

## schema flexibility 与 schema-on-read

`document database` 常被说成 `schemaless`，但这个说法容易误导。更准确的说法是 `schema-on-read`：数据库不强制每个 document 必须符合某个固定 schema，但读取这些 document 的应用代码通常仍然假设某种结构存在。

`relational database` 更接近 `schema-on-write`：schema 是显式的，写入时数据库会检查 row 是否符合 table definition。

这两者在 schema evolution 时差异明显。假设原来 user 只有一个 `name` 字段，现在要拆成 `first_name` 和 `last_name`。

在 `document database` 中，可以从新写入的 document 开始使用新字段；读取旧 document 时，应用代码负责兼容旧格式：

```javascript
if (user && user.name && !user.first_name) {
  user.first_name = user.name.split(" ")[0];
}
```

这让写入端迁移很灵活，但代价是所有读取路径都要记得处理旧格式。

在 `schema-on-write` 中，通常会执行 `schema migration`：

```sql
ALTER TABLE users ADD COLUMN first_name text DEFAULT NULL;
UPDATE users SET first_name = split_part(name, ' ', 1);
```

这让数据结构更统一，但大表上的 migration 可能有操作成本。现实里也可以折中：先快速添加 nullable column，再在读取或后台任务中逐步回填。

`schema-on-read` 适合数据结构异构、变化快、外部系统控制格式的场景；`schema-on-write` 适合同类记录结构稳定、需要约束和一致性的场景。

> [!NOTE] Wenbo 注
> 可以粗略理解为：SQL / `relational database` 在写入时更“严格”，MongoDB / `document database` 在写入时更“宽松”。但这里的严格和宽松主要说的是 schema enforcement，不是说 SQL 语法本身难不难。
>
> 在 `relational database` 中，`table` 先定义好有哪些 `column`、每个 `column` 是什么 type、哪些字段可以为 `NULL`、哪些字段必须满足 constraint。写入一行时，如果字段不存在、type 不对、违反 `NOT NULL` 或 `foreign key`，数据库会直接拒绝。这就是 `schema-on-write`：写入时就把数据形状管住。
>
> MongoDB 这类 `document database` 默认更宽松：同一个 collection 里的 document 可以有不同字段，也可以后来再给新 document 增加 `first_name`，旧 document 仍然只有 `name`。数据库未必阻止这种不一致。但读取时 application code 还是会假设某些字段存在，所以“宽松”的代价是读代码要处理旧格式、缺字段、字段类型不一致等情况。这就是 `schema-on-read`。
>
> 如果新版本代码按新 schema 读取老 document，而老 document 缺少某个字段，MongoDB 通常不会报错说“schema 不匹配”；它只是返回这个 document，缺的字段就是不存在。接下来会不会出错，取决于 application code。比如代码直接访问 `user.first_name`，可能得到 `undefined` / `null`；如果代码继续对它调用字符串方法，就可能在 application 层报错。
>
> 所以 document database 的 schema evolution 通常需要在读取时做兼容：如果 `first_name` 不存在，就从旧字段 `name` 临时推导；如果某个 array 不存在，就当成 empty array；如果字段 type 可能不同，就先做 type check。也可以在后台跑 migration/backfill，把旧 documents 慢慢改成新格式。
>
> 原文里提到给 SQL table 加一列时，常见做法是先 `ADD COLUMN first_name text DEFAULT NULL`。这里关键是 `NULL`：如果新列允许为 `NULL`，老 row 不需要立刻都有真实值，数据库通常可以很快完成这个 schema change。之后可以慢慢 backfill，把旧的 `name` 拆成 `first_name`。如果一上来要求 `NOT NULL` 且必须填好所有旧数据，就可能要重写大量 row，migration 成本更高。
>
> 所以这里不是简单的“SQL 麻烦，MongoDB 方便”，而是 trade-off：SQL 把一致性和结构约束前置到写入时；MongoDB 把灵活性给了写入端，但把兼容旧数据的复杂度推给读取端和 application code。

## data locality 与 document 的边界

`document model` 的另一个优势是 `storage locality`。一个 document 通常作为连续的数据块存储。如果应用经常读取整个 document，比如渲染一个完整 profile，那么 locality 很好：一次读取就能拿到大部分所需数据。

但 locality 只在“经常读取 document 的大部分内容”时才是优势。如果 document 很大，而查询只需要其中一小部分，数据库仍可能需要加载整个 document，造成浪费。如果经常对大 document 的小字段做更新，也可能需要重写较大的数据块。因此 document 通常应该保持相对小，并避免把无限增长的集合塞进同一个 document。

locality 也不是 document database 独有。某些 relational database 或 wide-column model 也提供类似能力，比如把父子行 colocate 或用 `column family` 管理局部性。这里真正重要的是访问模式：经常一起读的数据适合放近，经常独立更新或独立查询的数据适合拆开。

## document query language 与 relational-document convergence

`relational database` 主要使用 SQL。`document database` 的查询方式更多样：有些主要按 primary key 读取，有些支持 `secondary index` 查询 document 内部字段，有些提供比较丰富的 query language。

MongoDB 的 `aggregation pipeline` 是 document query language 的例子。它用 JSON-like syntax 表达一串阶段，例如 `$match` 过滤、`$lookup` 做类似 join 的操作、`$group` 做 aggregation。比如统计每个月观察到多少鲨鱼，SQL 里会用 `WHERE`、`GROUP BY`、`sum`；MongoDB 里则可能用 `$match` 和 `$group`。两者表达能力有重叠，只是语法风格不同。

这也引出本节的结论：`relational database` 和 `document database` 正在 convergence。很多 relational database 增加了 JSON column、JSON query operator、对 JSON 内部字段建 index 的能力；很多 document database 也增加了 `join`、`secondary index` 和 declarative query language。

这种融合很自然，因为真实应用常常同时需要两种能力：一部分数据像 document，一部分数据需要 relationship 和 join。最终选择不该停留在“SQL vs NoSQL”的标签上，而应该看具体数据结构、访问模式和一致性需求。

## 建模判断

如果数据是 tree-like、一对少、经常整体读取、嵌套项不常被外部单独引用，`document model` 往往自然。如果数据有大量 many-to-one、many-to-many、双向查询、共享实体、约束和 transaction，`relational model` 往往自然。

`normalization` 和 `denormalization` 也不是绝对阵营。`normalization` 让事实有唯一权威位置，更新和一致性更容易；`denormalization` 复制事实或预计算查询结果，让读取更快。大系统通常混合使用：稳定且昂贵的查询结果可以 materialize，变化快且复制成本高的信息继续 normalized，在读取阶段通过 join 或 hydrating 补全。
