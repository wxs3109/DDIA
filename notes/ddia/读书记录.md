### Chapter 5 记录

- 明确了 backward compatibility 和 forward compatibility 的记忆方式：新代码读旧数据是 backward compatibility，旧代码读新数据是 forward compatibility。
- 解释了 Java `Serializable`、Python `pickle` 这类语言内置序列化为什么不适合跨语言、长期保存或对外交换。
- 展开了反序列化安全问题：不可信输入可能触发任意类实例化、回调逻辑和 gadget chain，因此不能随便反序列化外部数据。
- 梳理了 JSON/XML/CSV 的问题：JSON number 精度不明确、二进制数据需要 Base64、CSV 缺少 schema 且转义规则容易被手写 parser 搞错。
- 补充了 JSON Schema 的学习例子：数据 JSON 和 schema JSON 是两份文档；`type`、`properties`、`required`、`minimum`、`maximum`、`patternProperties`、`additionalProperties`、`if`、`then` 等是 schema 关键字。
- 用一组带注释的 JSON/JSON Schema 例子串起了大整数 ID、Base64 payload、端口范围、枚举、整数形式的 object key、开放/封闭内容模型和条件约束。
- 读到二进制编码时，对 MessagePack 做了拆解：`0x83`、`0xa8` 这类是 byte，既可能是类型/长度控制 byte，也可能是普通内容 byte；MessagePack 仍然会把字段名写进编码里。
- 读到 Protocol Buffers 时，补充了和 MessagePack 的对比：protobuf 不写字段名，只写 field tag和值；接收方必须依赖同一份 `.proto` schema 才能把 field 1 解释成 `user_name`。
- 补充了 `.proto` schema 在工程里怎么约定：通常放在代码仓库、共享 SDK 或 schema registry，通过代码生成工具生成各语言代码；真正的约定发生在双方使用同一份 schema/生成代码这一步。
- 解释了 `repeated` 字段：protobuf wire data 中可以重复出现同一个 field tag，解码器根据 schema 把这些值收集成列表。
